# M6：已取消任务保护

**背景：** 任务一旦进入 `cancelled`，系统应优先保护终态一致性。允许少量无害留痕，但禁止任何把任务重新推进回主链的动作。

---

## M6-1：已取消任务允许追加事后留痕

**场景描述：** 任务取消后，旧执行 session 可能仍需要补一条事后说明或留痕记录。系统应允许这类无害追加，但任务终态不能被改变。

**测试步骤：**

1. Publisher 发布任务并由 alice 认领
2. Publisher 取消任务，任务进入 `cancelled`
3. alice 在任务上下文中追加一条事后留痕，例如“已通知用户任务取消”
4. 验证追加成功
5. 验证任务状态仍保持 `cancelled`
6. 验证该追加不会让任务重新进入 `ready / handoff / reworking / executing`

---

## M6-2：已取消任务拒绝重新回到待处理

**场景描述：** 已取消任务不允许通过任何残留动作重新回到待认领池，否则会破坏终态约束。

**测试步骤：**

1. 发布任务并让 alice 认领
2. Publisher 取消任务
3. 模拟旧执行链路尝试把任务重新放回池子
4. 验证系统返回失败，原因明确指向“任务已取消”
5. 验证任务状态保持 `cancelled`

---

## M6-3：已取消任务拒绝完成

**场景描述：** 任务已经取消后，即使旧执行人晚到一步，也不能再把它标成完成。

**测试步骤：**

1. 发布任务并由 alice 认领
2. Publisher 取消任务
3. alice 再尝试触发完成动作
4. 验证系统拒绝该动作
5. 验证返回原因体现“已取消/不在可完成状态”
6. 验证任务仍保持 `cancelled`

---

## M6-4：已取消任务的留痕不影响看板分组

**场景描述：** 取消后的事后留痕可能会更新 `updatedAt`，但不能因此让前端把任务误判成活跃任务。

**测试步骤：**

1. 取消一个任务
2. 对它追加一条事后留痕
3. 刷新看板数据
4. 验证该任务仍只出现在历史/终态区域
5. 验证不会重新出现在待接手、执行中或风险雷达中
