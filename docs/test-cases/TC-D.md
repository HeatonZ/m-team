# M4：取消与终态保护

---

## M4-1：Publisher 取消执行中的任务

**场景描述：** 任务已被认领并处于执行中，Publisher 因业务变更决定取消。系统应立即终止任务链，后续不允许再被认领。

**测试步骤：**

1. Publisher 发布任务
2. Executor 认领后进入 `running + executing`
3. Publisher 调用取消动作，并给出原因“优先级调整”
4. 验证任务状态变为：
   - `status=cancelled`
   - `completedAt` 已记录
5. 再查询任务，确认不会回到 `ready / handoff / reworking`
6. 让其他 agent 尝试认领该任务
7. 验证认领失败

---

## M4-2：非 Publisher 无法取消任务

**场景描述：** 只有任务发布者可以取消任务，其他人无权中断任务链。

**测试步骤：**

1. boss 发布任务
2. Executor 正在执行
3. other_user 尝试取消任务
4. 验证取消失败，原因明确指向“不是任务发布者”
5. 验证任务仍保持原运行状态，执行人不变

---

## M4-3：未认领任务可直接取消

**场景描述：** 新任务还未被认领时，Publisher 可直接取消，不会进入执行态。

**测试步骤：**

1. Publisher 发布任务
2. 验证初始状态为 `pending + ready`
3. Publisher 立即取消任务
4. 验证任务进入 `cancelled`
5. 验证任务从未进入 `executing`

---

## M4-4：终态任务不可重复取消

**场景描述：** 已取消任务再次取消，系统应拒绝，避免终态被重复写入。

**测试步骤：**

1. 发布并取消一个任务
2. 再次调用取消动作
3. 验证返回失败，原因为终态不可重复取消
4. 验证任务最终状态保持不变

---

## M4-5：任务进入 handoff 后仍可取消

**场景描述：** 第一棒已完成并交接，但 Publisher 在下一棒开始前决定终止任务，系统应允许取消，并保留已有 context。

**测试步骤：**

1. 第一棒执行完成，任务进入 `pending + handoff`
2. Publisher 调用取消动作
3. 验证任务变为 `cancelled`
4. 验证前序 context 历史仍完整保留
5. 验证后续 agent 无法再认领该任务
