# M5：回池与回收

---

## M5-1：Publisher 超时回收运行中任务

**场景描述：** 任务长时间处于运行中且未完成，Publisher heartbeat 触发超时回收，把任务放回池子，供后续重新处理。

**测试步骤：**

1. 发布任务并由 Executor 认领
2. 让任务持续运行超过超时阈值
3. Publisher heartbeat 检测到该任务超时
4. Publisher 调用回收动作
5. 验证任务被放回池子后：
   - `status=pending`
   - phase 回到可再次接手的状态
6. 验证后续 agent 可以重新认领

---

## M5-2：非当前执行人不能主动放回运行中任务

**场景描述：** 不是当前执行人的 agent，不能以自己的身份把别人的运行中任务放回池子。

**测试步骤：**

1. alice 认领任务并进入执行中
2. bob 尝试对该任务执行回池动作
3. 验证返回失败，原因为不是当前执行人
4. 验证任务仍由 alice 持有

---

## M5-3：已取消任务不能再回池

**场景描述：** 已取消任务属于终态，不应再被任何回池动作改回待处理状态。

**测试步骤：**

1. 发布任务并进入执行中
2. Publisher 取消任务
3. 再尝试执行回池动作
4. 验证返回失败，原因为任务已取消
5. 验证任务状态保持 `cancelled`

---

## M5-4：回池不会抹掉已有执行痕迹

**场景描述：** 任务被回收到池子后，已有上下文、lastExecutor 和历史记录必须保留，不能像新任务一样被洗白。

**测试步骤：**

1. 让任务先经历一段真实执行
2. 触发回池
3. 查询任务详情
4. 验证：
   - `lastExecutor` 仍存在
   - 既有 context 仍可见
   - 任务不是全新 ready 空白态
