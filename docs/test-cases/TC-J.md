# M10：排队与优先级

---

## M10-1：高优先级任务排在前面

**场景描述：** 待认领列表应按优先级排序，高优先级优先于普通和低优先级。

**测试步骤：**

1. 依次发布三个待认领任务：`normal`、`high`、`low`
2. 查询待认领列表
3. 验证返回顺序为：`high > normal > low`
4. 验证排序只针对可认领任务生效

---

## M10-2：同优先级按创建时间先进先出

**场景描述：** 相同优先级的任务，按创建时间升序返回，先创建的先被看到。

**测试步骤：**

1. 先发布任务 A，再发布任务 B，二者优先级相同
2. 查询待认领列表
3. 验证 A 排在 B 前面

---

## M10-3：不同 pending phase 仍遵守优先级顺序

**场景描述：** 即使任务都属于 pending，但 phase 分别是 `ready / handoff / reworking`，同一分组内仍应保持优先级和 FIFO 规则。

**测试步骤：**

1. 分别构造多个 `ready` 或多个 `handoff` 任务
2. 让它们优先级不同、创建时间不同
3. 查询待认领列表
4. 验证同类任务内部仍按优先级优先、同优先级 FIFO

---

## M10-4：running / finalizing 任务不混入待认领排序

**场景描述：** 运行中或收口中的任务不应混进待认领排序结果，否则会污染 heartbeat 的认领顺序。

**测试步骤：**

1. 构造若干 `pending` 任务与若干 `running/finalizing` 任务
2. 查询待认领列表
3. 验证结果只包含可认领的 pending phase 任务
4. 验证 running/finalizing 任务不会出现在待认领排序中
