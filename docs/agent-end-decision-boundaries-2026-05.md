# agent_end 判定边界（2026-05）

## 核心原则

`agent_end` 不能直接采信 executor 对“任务已完成”的自我结论。

它必须先从执行 transcript 中抽取三类事实，再做任务级裁决：

1. **当前步骤结论**
   - 当前 description 这一棒是否完成
   - 产出了什么文件 / 数据 / 结论
   - 是完整完成，还是只有中间结果

2. **当前步骤问题**
   - 是否存在未解决问题
   - 是否已经阻塞
   - 问题是否允许下一棒继续接手

3. **整个任务状态**
   - `goal` 是否已经满足
   - 是否还存在明确下一步
   - 当前应 `complete / relay / retain / fail`

一句话收敛：

> executor 提供事实；`agent_end` 做任务级判断。

---

## 四种裁决的最小标准

### 1. complete
只有在以下条件同时成立时才允许：
- 当前步骤已完成
- 有结构化结果或可验证产物
- 没有未解决问题阻止收口
- 没有明确下一步
- 从 transcript 看，**整个任务 goal 已满足**，而不是仅完成当前 description

### 2. relay
适用条件：
- 当前步骤已有有效结果
- 整个任务还没完成
- 下一步明确，且可以抽成单步 description 交给下一棒

### 3. retain
适用条件：
- 当前步骤有进展，但还不适合交接
- 或收尾信息不足，无法安全 complete / relay
- 或当前 executor 仍应继续收口

### 4. fail
适用条件：
- 当前步骤阻塞或无有效进展
- 无法形成可执行的下一步
- 问题已经使任务无法继续推进

---

## 必须避免的错误简化

### 错误 1：executor 说“完成”就 complete
错误原因：
- 这只能证明 executor 认为本步做完了
- 不能自动证明整个任务 goal 已满足

### 错误 2：只看当前步骤，不看整体 goal
错误原因：
- description 只是当前一棒
- 当前步骤完成 ≠ 整个任务完成

### 错误 3：只看整体 goal，不看当前问题
错误原因：
- 即使产物看起来接近终态，只要存在阻塞 / 缺口 / 未解决问题，也可能不能 complete

---

## 实现约束

`agent_end` 的判断顺序应接近：

1. 抽取当前步骤结果
2. 抽取当前步骤问题
3. 抽取下一步信息
4. 判断是否已满足整体 goal
5. 最终裁决 complete / relay / retain / fail

重点不是“最后一句话有没有写任务完成”，而是：
- 是否有证据证明本步完成
- 是否有证据证明整体完成
- 是否有问题阻断整体收口
- 是否存在明确下一步

---

## 给 executor 的边界提醒

executor 可以报告：
- 这一步完成了什么
- 产出了什么
- 遇到了什么问题
- 建议下一步做什么

executor 不负责决定：
- 任务是否 complete
- 是否应该 close
- 是否已经达到整条链的最终终态

这些由 `agent_end` 根据 transcript 独立判断。
