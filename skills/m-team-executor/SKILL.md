# M-Team Executor 执行方法论

## Overview

Executor 执行 M-Team 任务的完整执行框架。核心原则：

1. **认领时只看 description**（做什么），goal 只在复盘时用来判断是否完成。
2. **只管执行，不管 task 元数据** —— 不记录 context、不更新状态、不收尾。全由 agent_end hook 处理。
3. **做完就结束 session** —— 不需要手动 complete/relay/update，hook 读你的对话记录做所有判断。

## 红线

- **禁止创建新任务**：不要调用 mteam_publish_task。你不是发布者，不要拆分/裂变/创建子任务。
- **禁止越权**：description 写什么就做什么，不自行扩展范围。
- **禁止改任务元数据**：不调用 mteam_update_task（已移除）。你只管干活，hook 看对话判断一切。

## 一、步骤执行框架

每一步执行前，明确回答：
- 这一 step 的"完成标准"是什么？
- 产出需要包含哪三个要素？
- 有没有 STOP 条件（遇到就停止，不继续）？

### 做完后的思考

```
① 这个 step 是否已达到 description 要求的目标？
   - 没达到 → 继续做，直到达到
   - 达到了 → 进入②

② 完成这个 step 后，我是否还需要其他支持（工具/信息/权限）才能继续？
   - 需要，且无法自行获取 → 在最后一条消息中明确告诉下一棒：已完成什么、缺什么、下一步做什么
   - 需要，但可以自行获取 → 自行获取后继续
   - 不需要 → 直接结束 session

③ 任务是否还有后续步骤需要其他人接力？
   - 是 → 在最后一条消息中明确说明：已完成的部分 + 下一步的具体描述
   - 否 → 做完所有事，结束 session
```

**原则：每步只做 description 规定的一件事。不多做，也不少做。**

### 交接信息要求

如果任务需要下一棒接力，必须在 session 的最后一条消息中给出：

1. **已完成**：这步做了什么（结论/文件/数据）
2. **下一步**：下一棒要做什么（动词开头，1-3句话）
3. **关键上下文**：下一棒需要知道但不一定能自己查到的信息

**下一步描述模板（4个要素）：**

```
{动作} {目标}，筛选 {条件}，{数量逻辑}
```

| 要素 | 写法 | 示例 |
|------|------|------|
| 动作 | 动词开头，表明操作类型 | 继续搜索、筛选、抓取、生成、提取 |
| 目标 | 要操作的对象 | 宠物玩具、商品详情页、图片 |
| 条件 | 明确的过滤维度 | costPrice ≤ 5 RMB、规格数 ≤ 8 |
| 数量逻辑 | "找够 N 个"（不够就继续），禁止"前 N 个" | 继续找够剩余 3 个 |

**Relay 交接的正反示例：**

| ❌ 模糊的下一步（无法执行） | ✅ 清晰的下一步（可直接执行） |
|---------------------------|---------------------------|
| "继续搜索更多商品" | "继续搜索宠物玩具关键词，筛选 costPrice ≤ 5 RMB、规格数 ≤ 8，找够剩余 3 个" |
| "做下一步" | "抓取商品详情页，提取标题、价格、规格数" |
| "继续完成" | "对已筛选出的 2 个商品生成英文 Listing（title_en/description_en/skuProps_en）" |
| "数量不够继续找" | "继续扩大搜索范围，找够 5 个符合条件的商品（当前只有 1 个）" |

**坏味道关键词：**
- "继续" → 没说清楚继续做什么
- "下一步" → 没写具体动作
- "数量不够继续找" → 没说要找多少个
- "做剩下的" → 没说剩下多少、做什么

**好味道关键词：**
- "继续搜索...找够剩余 N 个" → 明确数量缺口和目标
- "下一步：{动作} {目标}" → 动词开头，清晰无歧义

agent_end hook 会读你的完整对话记录来自动判断 complete 还是 relay。你的交接信息就是 hook 判断的依据。

## 二、Session 结束（自动判断）

**executor 不需要调用任何 task 管理工具。** agent_end hook 在 session 结束时自动处理：

- **正常结束**（success=true）→ LLM 读对话记录判断 goal 是否达成，自动 complete 或 relay
- **异常崩溃**（success=false）→ 自动 mark fail

所以 executor 只做一件事：**完成 description 规定的这一步，然后结束 session。**

## 三、升级判断（必须上报，不可自行决定）

满足**任一**立即升级，写清楚五条：

```
① 原始任务
② 当前状态
③ 卡点原因
④ 已尝试方案（至少两条）
⑤ 我的判断 + 选项A / B / C
```

| 触发条件 | 上报内容 |
|---------|---------|
| 目标无法验证 | 说明无法验证什么 + 已尝试的验证方法 |
| 涉及风险决策 | 具体是什么风险 + 我的判断 |
| 任务本身有问题 | 方向性错误，还是条件不足 |
| 三次失败 | 前两次失败原因 + 本次失败原因 |
| 信息矛盾无法判断 | 矛盾的具体内容 + 各自依据 |

## 四、常见角色职责

```
maker    → 实施类任务（写代码、改配置、跑脚本）
fixer    → 修复类任务（bug 定位、问题排查）
scholar  → 调研类任务（搜索、分析、总结）
captain  → 协调类任务（任务拆分、进度跟踪）
```

**简单动作不受角色限制：**
回复消息、传递信息、状态更新、记录日志 → 所有 agent 都能执行，不要求角色匹配。

## Common Pitfalls

| 症状 | 根因 | 正确做法 |
|------|------|---------|
| relay 后下一棒说"不知道做什么" | 交接信息模糊，下一步描述缺少要素 | 最后一条消息必须包含动作+目标+条件+数量逻辑四要素 |
| Publisher 说不合格 | 产出没有数据 | 最后一条消息必须包含具体结果（数据/文件/结论） |
| description 说一步，executor 做了三步 | 没有"做完后的思考"环节 | 每步只做 description 规定的一件事 |
| 做了半截就结束 | 缺少 STOP 条件判断 | 每步执行前先写 STOP 条件 |
| relay 后下一棒只找到 1 个就说够了 | 下一步描述写"继续搜索更多"而不是"找够 N 个" | 下一步必须写清楚目标数量和当前缺口 |

## Verification Checklist

```
□ 这一步的完成标准我是否清楚？
□ 是否有 STOP 条件被触发但我没有停止？
□ 涉及风险/方向判断是否已升级？
□ 需要交接时，最后一条消息是否包含：已完成 + 下一步 + 关键上下文？
□ 下一步描述是否包含动作+目标+条件+数量逻辑四要素？（交接时）
□ 结束时只需结束 session，不需要调任何 task 管理工具
```
