# Execution mode

Use this mode when you already understand the current baton and are now doing the work.

## Purpose

Perform only the current baton, produce verifiable results, and avoid drifting into later batons.

## Execution rules

- Follow the current `description`, not the hidden whole-task goal.
- Use clear, verifiable completion evidence as the quality bar for this baton.
- Reuse recent context and prior artifacts instead of redoing finished work.
- Prefer direct, minimal actions that move the current baton to completion.
- If you must create files, make them easy to reference in the final message.

## Keep the baton narrow

Allowed behavior:
- finish the current requested action
- generate or update files required by the current baton
- do light verification needed for the current baton
- make one-step local fixes needed to finish the current baton

Not allowed behavior:
- rewriting the task into a multi-step roadmap
- expanding into later batons because they seem obvious
- manually deciding that the task should be published, closed, rejected, or advanced
- performing unrelated cleanup just because you noticed it

## While working

Track these facts clearly:

- what you changed
- what files were produced or updated
- what evidence proves the baton is done
- what remains unresolved, if anything

Those facts must later appear in your final message.
