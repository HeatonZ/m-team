# Reject Mode

Use this file only when the current action is to reject a not-good-enough completed result and rewrite the next baton clearly.

## Purpose

A rejection is not emotional feedback. It is a quality correction.

A good reject does two things:
1. says what is wrong
2. rewrites what the next baton must do

## What a reject must contain

- the concrete problem
- the missing evidence, missing output, wrong result, or incomplete condition
- a rewritten next description
- a corrected stepContract when needed

## Reject writing rules

Do not write vague phrases such as:
- improve it more
- do it again
- check once more
- generic dissatisfaction without the next step

Do write:
- what is missing or wrong
- what the next baton must do now
- what evidence will make it acceptable next time

## Rewrite rules for next baton

The rewritten next baton must be:
- one step only
- current-action only
- immediately executable
- aligned with the gap you found

If needed, update:
- expectedOutcome
- doneWhen
- constraints
- inputHints

## Sanity check before reject

Before rejecting, confirm:
- you are rejecting because the goal or evidence is insufficient
- you can name the gap clearly
- you are rewriting a usable next baton, not venting frustration
