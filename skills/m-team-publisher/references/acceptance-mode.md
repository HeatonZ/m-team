# Acceptance Mode

Use this file only when the current action is to review a completed task and decide whether it truly satisfies the goal.

## Purpose

Decide whether a completed task should:
- be accepted as good enough for closure
- be rejected and rewritten into a better next baton

Do not think about how to publish a fresh task while you are accepting.

## What to inspect

Use the current runtime truth:
- goal
- current task description
- context
- produced files
- latest step evidence

Acceptance is not based on effort or intent. It is based on facts.

## Acceptance checks

Check all of the following:
1. The final goal is actually satisfied
2. Real evidence exists in context or files
3. The latest completed result is consistent with the goal
4. The task is truly final and not only one baton of a still-incomplete chain
5. No meaningful unresolved quality gap remains

## Accept only when all are true

- goal is satisfied
- evidence is sufficient
- no meaningful unresolved gap remains
- closing now would not hide unfinished work

## Do not accept just because

- the executor worked hard
- the summary sounds confident
- some files exist but they do not prove the result
- one baton was done but the full goal is still unmet

## If not acceptable

Do not close.
Move to reject mode and rewrite the next baton clearly.
Read references/reject-mode.md.
